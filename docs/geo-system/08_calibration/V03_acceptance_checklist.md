# V03. Приёмка версий базы и промптов: чек-лист против системных уклонов

**Слой:** методология процесса. Операционализация Принципов 9–12 рефлексии ИНСТ-проекта: каждая доработка проходит эту приёмку, потому что известные уклоны генеративной сборки (пересборка вместо инкремента, декларативная полнота, скрытая компрессия, зашивание состояния) без формальной процедуры воспроизводятся. Приёмщик — заказчик-рецензент; исполнитель обязан подготовить материалы приёмки САМ, до передачи.

## 1. Что считается «версией»
Любое изменение любого файла базы или промптов. Классы: **мажорная** (новые/переработанные файлы, изменение шкал/якорей/процедур), **минорная** (дополнение без изменения существующего смысла), **фактологическая** (обновление дат/чисел/статусов). Класс определяет глубину приёмки (§3–§6 — для мажорных полностью; минорные — §3, §5; фактологические — §5).

## 2. Обязательный пакет исполнителя (без него приёмка не начинается)
(а) **Опись наследования** (Принцип 9): таблица «элемент старой версии → куда вошёл в новой / изменён как / удалён почему-с-санкции-заказчика». Молчаливое удаление = автоматический возврат версии без чтения. (б) **Диф-резюме**: что добавлено/изменено по файлам, одним экраном. (в) **Заявление полноты**: список всего, что версия ОБЪЯВЛЯЕТ существующим (файлы, кейсы, шкалы, разделы). (г) Обновлённый манифест README со статусами.

## 3. Проверка полноты содержанием (Принцип 10)
Для каждого пункта заявления полноты — вскрытие: файл существует, развёрнут по своему формату (кейсы — 9 разделов; теория — 5-компонентный шаблон; шкалы — якоря на каждом делении), не является оглавлением/заглушкой/«см. таблицу». Выборочно — 100% новых элементов, ≥20% старых (регресс-контроль). Красные маркеры: разделы из одних заголовков; «будет добавлено»; ссылки на несуществующие файлы; кейс без раздела «уроки для шкал».

## 4. Проверка против скрытой компрессии (Принцип 12)
Сравнение объёмов пофайлово: сокращение файла при содержательной доработке — флаг, требующий явного объяснения в описи («ужато X ради Y с сохранением…»). Правило: рост требований не оплачивается молчаливым выбрасыванием; при реальной необходимости сокращения — отдельная санкция заказчика.

## 5. Проверка эпистемической гигиены (Принцип 5) и слоёв (Принцип 1)
Выборочный проход: (а) все числа/даты несут метки [ФАКТ]/[ОЦЕНКА]/[поиск]/[гипотеза] и источник класса A/B там, где двигаются баллы; (б) в базу не зашито состояние (текущие баллы, вероятности, «актуальные» уровни) вне явно помеченных снимков-примеров с датой; (в) конфликтные данные не «замазаны» выбором одного числа; (г) отсутствие данных нигде не превращено в отсутствие проблемы.

## 6. Проверка согласованности системы
Автоматизируемая часть (скриптом или проходом): уникальность и взаимная согласованность кодов (G1–G13, E1–E15, S1–S4, W1–W8, K01–K32+, T/C/B/V/D-файлы); все внутренние ссылки разрешаются; якоря шкал ссылаются на существующие кейсы и не противоречат их фактам; изменение любого якоря/веса сопровождается записью «было→стало→основание» (V01 §5); нулевые точки и снимки датированы; kb_version/commit обновлён и процитирован в README.

## 7. Формат замечаний и цикл
Замечания заказчика классифицируются: **блокирующее** (фактическая ошибка, потеря контента, нарушение принципа — версия возвращается), **содержательное** (глубина/точность — исправление в этой версии), **очередь** (новые идеи — в манифест следующей). Исправление блокирующих и содержательных сопровождается мини-описью «замечание → что изменено → где». Версия принята = запись в журнале версий (дата, класс, диф-резюме, замечания и их судьба) — журнал сам является файлом базы и наследуется по Принципу 9.

## 8. Специальные проверки по типам артефактов
**Промпты:** тест «читателя с нуля» — агент, видящий только промпт+базу, не имеет пространства ложной интерпретации (Принцип 2: у каждой концепции — алгоритм, пример, анти-паттерны, шаблон вывода); запрет зашитых баллов — проверка поиском числовых оценок. **Кейсы:** сверка хронологии с первичными источниками по выборке; наличие ex-ante теста (без него кейс — журналистика, не калибровочный материал). **Шкалы:** каждое деление различимо от соседних по наблюдаемым критериям; «за-якорем» правило (B01 §правила) не нарушено. **Сценарии/лестницы:** маркеры наблюдаемы ex ante, вероятности не зашиты.

## 9. Анти-паттерны самой приёмки
Приёмка по оглавлению (читать содержание, не заголовки — весь смысл §3). «Доверяй исполнителю» на описи (опись сверяется выборочно с реальными дифами). Приёмка без доменной экспертизы (уроки рефлексии: все шесть провалов ИНСТ-проекта поймал только придирчивый доменный читатель — этот цикл не оптимизация, а часть архитектуры). Замечания без классификации (превращают цикл в бесконечный). И зеркальный: перфекционизм по «очереди» — блокировать версию можно только блокирующими.
