## Продукт
Basis — независимый аналитический слой и «второе мнение» для частного инвестора
(не брокер, не сделки, не «купить/продать»; снижает неопределённость, не хайп).
3 модуля: Рынок (карточки компаний + классы активов), Обозреватель, Портфель.
Стек: React (CRA, чистый JS, `/frontend/Basis`, UI в `src/App.js`),
FastAPI+PostgreSQL+SQLAlchemy+Alembic (`/backend`), деплой git push → Timeweb.
Данные деплоятся на бой автоматически при старте (идемпотентно), без ручных команд.

## 🔴 ГОТОВО = РАБОТАЕТ НА ЖИВОМ САЙТЕ
«Код запушен» ≠ готово. Готово = открыл inbasis.ru и УВИДЕЛ результат: данные в
БД → бэк отдаёт (проверь БОЕВОЙ адрес nikitasoin-basis-e726.twc1.net, не
localhost) → фронт рендерит → задеплоено. Деплой не мгновенный: жди смену бандла
`main.<hash>.js` / опрашивай боевой эндпоинт (до 10 мин), не суди «сломано» сразу.
НИКОГДА не отчитывайся «сделано», если на живом сайте этого нет.
«Готово» по аналитической задаче = сделано ПО МЕТОДИКЕ (включая качественные
блоки), а не «поля заполнены числами». Частично — честно скажи, что да, что нет.

## 🔴 ОТК — СМОТРИ НА ЭКРАН ГЛАЗАМИ, А НЕ НА ЗАМЫСЕЛ
Перед готовностью прогони ОТК по тому, КАК ВЫГЛЯДИТ НА ЖИВОМ САЙТЕ:
- `user-persona-reviewer` — понятно ли, ЧИТАЕТСЯ ли (шрифты, каша, «простыня»),
  отвечает ли «что мне с этим делать». ВЕСЬ экран целиком + сверка с соседними
  вкладками (иконки, переходы, заполненность, единые термины).
- `basis-design-reviewer` — дизайн-конституция (для UI).
Мнение персоны оценивай критично (неполный обзор — не ломай правильное), но
РЕАЛЬНЫЕ дефекты (нечитаемо, простыня, прочерки) чини САМ до возврата.

## 🔴 НЕ ОСТАНАВЛИВАЙСЯ — СООБЩЕНИЕ ВЛАДЕЛЬЦУ = КОНЕЦ РАБОТЫ
Написать владельцу отчёт = завершить ход и встать. Поэтому:
- **Промежуточные чекпойнты пиши ТОЛЬКО в `docs/work-journal.md`** (что сделано,
  что ждёт ОК, что осталось) — и РАБОТАЙ ДАЛЬШЕ. НЕ пиши промежуточных отчётов
  в чат: захотел отчитаться — это сигнал записать в журнал и продолжить.
- Сообщение владельцу в чат пиши ТОЛЬКО когда: (а) очередь исчерпана / всё, что
  можно без него, сделано; (б) блокирующая развилка, без которой дальше никак;
  (в) технический предел сессии (контекст). Тогда — сводный отчёт из журнала.
- Развилки НЕ блокирующего характера копи в журнале и продолжай очередь.
- UX-баги всего сайта (потеря состояния списков, несогласованные переходы) —
  приоритетнее нового контента.
- Лучше НОВОЕ вчерне, чем простаивать; «вчерне» ≠ без ОТК и ≠ нечитаемо.
- Большие раскатки (сотни единиц) — батчами, коммит после каждого батча, статус
  в журнал, СЛЕДУЮЩИЙ БАТЧ СРАЗУ. Несколько сессий — норма: новая сессия читает
  журнал и продолжает с места обрыва без вопросов.

### Очередь задач (бери сверху вниз; детали в docs/status.md)
1. ФИКС спредов в bond_risk.py: YTM→YTP к ближайшей пут-оферте; спред флоатера —
   к КС/RUONIA, не к G-кривой; неликвид — без вердикта. (Ложный спред МТС/AAA с
   YTM 64% — артефакт near-offer.) Пересчитать затронутые, потом продолжать
   раскатку — иначе остальные эмитенты посчитаются на кривых числах.
2. Облигации вглубь: продолжение раскатки «доходность vs риск» по методичке v1.1
   (~356 эмитентов, батчами по ~12, с реальными блоками A–F и веб-поиском).
3. ПЕРЕПРОГОН ФИНАНСОВ карточек компаний по financial-analyst v2 (нормализация
   reported/adjusted с мостом, налог 25%, исторический P/B, маршрутизация методов
   по сектору, explain «как посчитано»). Порядок: сначала компании с крупными
   разовыми факторами и аномалиями (Лукойл, ВТБ, Газпром, banks, кубышки), затем
   по секторам. Витрина: показать adjusted рядом с reported + мост + «как
   посчитана справедливая цена» по каждому методу.
4. Фьючерсы вглубь: вкладка «Анализ базового актива» (акция → выжимка из карточки
   + ссылка; иной актив → рынок + гео).
5. Валюта и металлы: платина/палладий, больше пар, читаемый анализ.
6. Обозреватель рынка (5 под-блоков — status.md).
7. Ревизия карточек (fact-checker + читаемость; вкладки не пересобирать).
8. Скрининг (углубить); Лендинг (глубокий, маркетинговый).
Опционы — НЕ трогать без отдельного ОК владельца.

## 🎨 ДИЗАЙН-СИСТЕМА — единый источник правды: `docs/design-system.md`
Канон — **NEO (Basis Neo-Institutional Research System)**: Source Serif 4 (заголовки/проза),
JetBrains Mono (числа), Inter (UI); темы light «Research Desk» / dark «Command Center» через
алиасинг токенов в `.cc-root` (классические имена → `--cc-*`). Классика — ЛЕГАСИ, только за
`?classic=1`. Все значения/компоненты — в `docs/design-system.md` (читать перед любым UI).
Всегда `var(--token)`, хардкод hex нельзя. **Новый/переделанный экран ОБЯЗАН:** «4 слоя чтения»
(что смотрю → сигнал → доказательство+надёжность → действие; вердикт поверх данных) +
эпистемические теги (факт/оценка/модель/суждение). Голая таблица без интерпретации ≠ готовый экран.

## ⚙️ Технические правила (кратко; подробности в playbook)
- Backend-фича: модель → схема → сервис → эндпоинт → тест.
- Схема БД — только Alembic-миграции. Массовая закачка — скриптом.
- Секреты — только `.env`, НИКОГДА не хардкодить.
- Простые решения. Три уровня достоверности (факт/оценка/суждение) во всей
  аналитике. Без «купить/продать».
- MOEX ISS и веб — последовательно с паузами, батчами (иначе rate limit).
- «rate limit (not your usage limit)» = повисшая сессия/плотный поток: новая
  сессия, паузы. Новая сессия начинает с чтения docs/work-journal.md.
- Объясняй владельцу как НЕ-технарю: видимый эффект → причина → деталь.
