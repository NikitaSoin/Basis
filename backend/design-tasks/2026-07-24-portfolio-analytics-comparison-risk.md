# Дизайн-задача: «Аналитика портфеля» — Сравнение / Доходность и оценка / Риски

Дата: 2026-07-24
Компонент: `frontend/Basis/src/portfolio/PortfolioViews.jsx` (компонент `PortfolioV2`)
Бэкенд уже реализован и протестирован end-to-end (см. «Контракт данных» ниже) — здесь
только фронтенд. Инициатива владельца: продуктовое ревью блока сравнения портфеля с
бенчмарками + методологический фикс CAPM + новые риск-метрики.

## Контекст и почему это делаем

Владелец (2026-07-23) посмотрел блок «Аналитика портфеля» и поднял пять проблем:
1. В «Сравнение» нет выбора временного интервала — окно фиксировано.
2. Не объяснено, ПОЧЕМУ сравниваем именно с MCFTR/IMOEX — голые тикеры непонятны
   рядовому инвестору («индекс МосБиржи полной доходности» не всем известен).
3. Хочется больше бенчмарков — отраслевые индексы, смешанный бенчмарк «по весам
   портфеля».
4. В таблице «Доходность и оценка» колонка «Ваша доходность» непонятна и не нужна;
   CAPM считался от исторической доходности индекса вместо ERP Дамодарана.
5. В «Рисках» нужен VaR 99% (переключатель с 95%), CVaR/Expected Shortfall, и
   продуктовое объяснение ценности этих метрик (какую «джобу» они выполняют).

Отдельно найден и уже исправлен бэкенд-баг: колонка «Ваша доходность» физически
никогда не доходила до фронта (Pydantic-схема молча резала недекларированное поле
`your_return_pct` — классический для этого проекта класс бага, см.
`.claude/agents/../MEMORY` про Decimal/isinstance). Значит на бою эта колонка ВСЕГДА
показывала «—» — просто убираем мёртвый код, ничего у пользователя не «отнимается».

## Контракт данных (эндпоинт `GET /api/portfolios/{id}/metrics?period=...`)

`period` — query-параметр, один из: `1m`, `3m`, `6m`, `1y`, `3y`, `max` (default `3y`).
Влияет ТОЛЬКО на график «Сравнение» (`benchmark`); волатильность/бета/VaR/корреляции
всегда считаются на фиксированном 3-летнем окне независимо от периода.

### `benchmark` (объект, было — расширено)
```
{
  dates: string[],              // ISO-даты, прорежены до ~260 точек
  portfolio: number[],          // накопленная % доходность портфеля от начала ВЫБРАННОГО периода
  mcftr: number[],               // индекс МосБиржи полной доходности (с дивидендами), от начала периода
  imoex: number[],               // индекс МосБиржи ценовой (без дивидендов), от начала периода
  sector_blend: (number|null)[] | null,   // НОВОЕ: смешанный бенчмарк по весам секторов портфеля, от начала периода
  sector_blend_coverage_pct: number | null,  // НОВОЕ: % equity-стоимости портфеля, покрытой секторными индексами
  sector_blend_covered_sectors: string[],    // НОВОЕ: какие секторы вошли (напр. ["Нефть и газ", "Финансы"])
  sector_blend_excluded_sectors: string[],   // НОВОЕ: какие НЕ вошли (нет индекса или <90% покрытия дат)
  period: string,                 // НОВОЕ: эхо выбранного периода ("1m".."max")
  period_years: number,
  limited_by: string | null,       // тикер самой молодой бумаги, если она ограничивает окно
  portfolio_total_pct: number | null,
  benchmark_total_pct: number | null,   // итог MCFTR за период
  sector_blend_total_pct: number | null,  // НОВОЕ: итог смешанного бенчмарка за период
  note: string,
}
```
Если `sector_blend_covered_sectors` пуст (портфель весь в секторах без индекса —
Машиностроение/Здравоохранение/Прочее) — `sector_blend` будет `null`, это ожидаемо,
рендерить нужно только когда `sector_blend` не `null`.

### `positions[]` (было — добавлены поля)
```
upside_to_fair_pct: number | null,   // (справедливая цена база − текущая цена) / текущая цена, %. ТОЛЬКО equity
valuation_flag: "undervalued" | "fair" | "overvalued" | null,   // наш порог: >+15%/-15%/между
fair_value_as_of: string | null,     // дата анализа справедливой цены (может быть многомесячной давности!)
var_99: number | null,                // дневной VaR 99%, % потери (положительное число)
cvar_95: number | null,               // дневной CVaR/Expected Shortfall 95%, % потери
cvar_99: number | null,
var_95_annual: number | null,         // ГОДОВОЙ VaR (перекрывающиеся 252-дневные окна), % потери
cvar_95_annual: number | null,
var_99_annual: number | null,
cvar_99_annual: number | null,
price_as_of: string | null,           // non-equity (bond/fund): дата цены — ПОКАЗЫВАТЬ ВСЕГДА для bond/fund
```
Поле `your_return_pct` — БОЛЬШЕ НЕ СУЩЕСТВУЕТ (ни в positions, ни в portfolio). Не
пытайтесь его читать.

### `portfolio` (агрегат — добавлены поля)
Те же новые поля, что в positions (`var_99`, `cvar_95`, `cvar_99`, `var_95_annual`,
`cvar_95_annual`, `var_99_annual`, `cvar_99_annual`), плюс:
```
upside_to_fair_pct: { value: number|null, n: number, m: number } | null   // средневзвешенный, "по n из m позиций"
```

### `rates` (было — изменена семантика!)
```
risk_free_1y: number | null,     // ОФЗ ~1г — теперь ТОЛЬКО контекст/для облигаций, НЕ вход в CAPM
risk_free_10y: number | null,    // НОВОЕ: ОФЗ ~10л — РЕАЛЬНЫЙ Rf в CAPM/Sharpe/Sortino/альфа (= карточка компании)
risk_free_as_of: string | null,  // теперь дата ОФЗ-10л
erp_pct: number | null,          // НОВОЕ: ERP Дамодарана (mature ERP + country risk premium РФ) — реальный вход в CAPM
market_return_3y: number | null, // CAGR MCFTR за 3г — ТЕПЕРЬ ТОЛЬКО факт-контекст ("рынок реально заработал"), БОЛЬШЕ НЕ используется в формуле CAPM
market_premium: number | null,   // факт: Rm(MCFTR,3г) − Rf(1г) — тоже контекст, не вход в CAPM
```
**Это критично для копирайта**: везде, где текст говорит «Rf — ОФЗ ~1 год» или
формула CAPM написана как `Rf + β×(Rm−Rf)`, — это теперь НЕВЕРНО и должно быть
переписано (см. список ниже).

## Что переписать — построчный список (важно не пропустить ни одного)

Все найдены грепом `risk_free_1y|ОФЗ ~1|Rm − Rf|market_return_3y` — их ровно столько,
больше нигде в файле CAPM/Rf-формулы не упоминаются:

1. **`RETURN_COLUMNS` (~строка 1590–1627)**: удалить объект колонки `yourReturn`
   (ключ `"yourReturn"`, label «Ваша доходность») целиком. Колонку CAPM (`key: "capm"`)
   — поменять tooltip с `"Модельная forward-оценка: Rf + β×(Rm − Rf)..."` на что-то
   вроде `"Модельная forward-оценка: Rf(ОФЗ 10л) + β×ERP(Дамодаран). Оценка, не факт и не прогноз"`.
   Добавить новую колонку сразу после CAPM (или после `return3y`) — «Апсайд к
   справедливой цене» (`upside_to_fair_pct`), ТОЛЬКО для equity (для non-equity —
   «—»), С эпистемическим тегом «суждение» (как на карточке компании, вкладка
   «Финансы», блок «Справедливая стоимость» — используй тот же тег/стиль, см.
   `FinanceTab.jsx` класс `tag-judg`/`суждение`), и с раскрытием даты анализа
   (`fair_value_as_of`) в тултипе — что-то вроде: `Апсайд к справедливой цене
   (аналитика от {fair_value_as_of}): рынок vs наша модель`. Рядом/вместо — небольшой
   цветной индикатор `valuation_flag` (undervalued → зелёный «Недооценена», fair →
   нейтральный «Справедливо», overvalued → предупреждающий «Переоценена»), монохромная
   иконка, цвет только в самом значении (дизайн-конституция).
2. **`METRIC_EXPLANATIONS.capm.formula`** (~строка 1285): переписать `expr` на
   `"Ожидаемая доходность = Rf + β × ERP"`, `note` — что Rf теперь ОФЗ ~10 лет (тот же
   вход, что и в оценке справедливой стоимости на карточке компании — методологии
   СВЕДЕНЫ), ERP — премия за риск рынка акций РФ по Дамодарану (mature market ERP +
   страновая премия), НЕ историческая доходность индекса. Убрать фразу «В текущем
   периоде премия отрицательна, поэтому модельные оценки занижены» — она была верна
   для старой формулы (Rm−Rf могла быть отрицательной), с ERP-константой такого не
   бывает.
3. **`METRIC_EXPLANATIONS.sharpe.formula`** (~строка 1406) и **`.alpha.formula`**
   (~строка 1418): заменить «Безрисковая ставка — ОФЗ ~1 год» на «ОФЗ ~10 лет»;
   `alpha.formula.expr` — заменить `Rf + β × (Rm − Rf)` на `Rf + β × ERP`.
4. **Пояснительный блок под таблицей рисков** (~строка 2906–2913, внутри `pfRisk`):
   сейчас пишет «Безрисковая ставка: ОФЗ ~1 г ... Рынок (MCFTR, 3 г): ...; премия:
   ...» — читается как «вот что идёт в модель». Переписать на разделение факт vs
   модель: например — *«Безрисковая ставка в моделях (CAPM/Шарп/Сортино/альфа): ОФЗ
   ~10 лет {rates.risk_free_10y}% на {rates.risk_free_as_of}. Премия за риск рынка
   акций (ERP, по Дамодарану): {rates.erp_pct}%. Для контекста: рынок (MCFTR)
   фактически заработал за 3 года {rates.market_return_3y}% — это НЕ то же самое, что
   ERP в модели (историческая доходность индекса — шумный ориентир, ERP — структурная
   оценка)»*. Раздели визуально факт-контекст и модельный вход (два разных цвета
   текста или ячейки), чтобы не сливались.
5. **Заголовок CAPM-колонки таблицы** (~строка 1615, tooltip у `capm` в
   `RETURN_COLUMNS`) — уже покрыто пунктом 1.
6. **`explainCtx`** (~строка 2285: `rf: pfMetrics?.rates?.risk_free_1y ?? null`) —
   этот `rf` идёт в `ctx.rf` для `MetricExplainers` (используется в `capm.reading` и
   `earnings_yield.reading` для сравнения с безрисковой ставкой). Замени на
   `pfMetrics?.rates?.risk_free_10y ?? null` — иначе объяснения CAPM будут сравнивать
   с неправильной (короткой) ставкой.
7. **`RISK_REGIME_NOTE`** (~строка 1435) — упоминает «ОФЗ держится высоко — около
   12–13%» и относится к ОБЩЕЙ логике «высокая ставка → слабый Шарп/Сортино по всему
   рынку» — эта логика по сути не изменилась (Rf всё ещё высокий, просто теперь 10л, а
   не 1г), можно оставить как есть ИЛИ уточнить цифру, если легко проверить актуальную
   — не критично, не обязательная правка.

## Новое: VaR 99% / CVaR — таблица + объяснения

В `RISK_COLUMNS` (~строка 1629) сейчас одна колонка `var95` (VaR 95%, всегда
видна). Нужно:
- Добавить переключатель уровня доверия **95% / 99%** для ВСЕЙ группы риска (тумблер
  над таблицей, в духе уже существующих UI-паттернов Basis — не изобретай новый
  визуальный язык, посмотри `.bs-chip`/переключатели в `docs/portfolio-analytics.html`
  если есть, иначе — простая пара кнопок-таб). При переключении колонка `var95`
  показывает `var_99` вместо `var_95` (используй тот же рендер, меняй ключ и label).
- Рядом с VaR добавить колонку/значение **CVaR** того же уровня (`cvar_95`/`cvar_99`
  синхронно с переключателем) — обязательно ПОДПИСАТЬ разницу («VaR — где граница
  плохого хвоста; CVaR — насколько плохо ВНУТРИ этого хвоста в среднем»), не просто
  два похожих числа рядом без объяснения разницы.
- Добавить **годовой** горизонт как ещё один переключатель или доп. строка/колонка
  (дневной vs годовой) — используя `var_95_annual`/`cvar_95_annual`/`var_99_annual`/
  `cvar_99_annual`. Не обязательно 4 переключателя сразу — можно 2 ортогональных тумблера
  (95%/99% × дневной/годовой) ИЛИ более простая форма — на твоё дизайнерское усмотрение,
  главное чтобы пользователь мог получить все 4 комбинации без перегрузки экрана.
- В группу `portfolio_row` (`pfRisk`, сводная строка «Портфель» — там же, где сейчас
  показывается `var95: pfMetrics?.portfolio?.var_95`) добавь то же самое для новых
  полей.
- **Продуктовая подача (важно, это была явная просьба владельца)**: переведи % в
  рубли на РЕАЛЬНОМ размере портфеля (`pfMetrics.portfolio.total_value`) — например
  под таблицей или в `KeyTakeaway`: *«Ваш портфель ₽{total_value}. В обычный плохой
  год (1 из 20) вы можете правдоподобно оказаться на уровне ≈₽{total_value×(1−var_95_annual/100)}.
  Если случится тот самый редкий (1 из 100) провал — глубина потерь в среднем внутри
  этого хвоста — ≈₽{total_value×(1−cvar_99_annual/100)}»* (числа ориентировочные,
  посчитай точно по формуле). Это прямая просьба владельца — «донести ценность метрик,
  какую джобу они выполняют» — голые проценты этого не делают, рубли на реальной сумме
  делают.
- Добавь новые записи в `METRIC_EXPLANATIONS` (по образцу уже существующего `var_95`,
  строка ~1350): `var_99`, `cvar_95`, `cvar_99` (и опционально annual-варианты, либо
  один explainer с пометкой «дневной/годовой» в `reading`). Используй тот же паттерн
  `type/what/reading/soWhat/formula`. Черновик содержания:
  - `var_99`: «Оценка настоящего плохого дня (1 к 100), а не рядового». soWhat: «99% —
    более редкий и глубокий хвост, чем 95%; используйте вместе, чтобы увидеть, как
    быстро растёт потенциальная потеря при переходе от „плохого“ к „очень плохому“».
  - `cvar_95`/`cvar_99`: «Не просто ГРАНИЦА плохого хвоста (это VaR), а средняя потеря
    ВНУТРИ него — отвечает на вопрос „а если я всё-таки попал в те самые 5%(1%),
    насколько плохо там внутри“». Это ключевое отличие от VaR, подчеркни его явно.
  Добавь эти ключи в `metricKeys` массив, который передаётся в `<MetricExplainers>`
  внутри `pfRisk` (~строка 2935: `metricKeys={["volatility", "var_95", ...]}` — допиши
  `"var_99", "cvar_95", "cvar_99"`), и соответствующие значения в `values={{...}}`
  (~строка 2936).
- Добавь explainer для `upside_to_fair_pct` (используется в панели «Доходность и
  оценка», не «Риски») — тег «суждение», объясни, что это апсайд ЖИВОЙ цены к
  синтезированной справедливой цене с карточки компании, дата анализа может быть
  многомесячной давности (прозрачность!), не переоценивай точность.

## Новое: интервал сравнения + человеческие бенчмарки + смешанный бенчмарк

В панели `pfCompare` (~строка 2656–2825):
1. **Кнопки периода** над графиком `BenchmarkChart`: 1М / 3М / 6М / 1Г / 3Г / Весь
   период (см. `_PERIOD_DAYS` бэка: `1m/3m/6m/1y/3y/max`). При клике — рефетч
   `/metrics?period=...` (или, если уже есть локальный кэш всех периодов — проще
   рефетчить каждый раз, данные лёгкие). Активная кнопка — акцент медь
   (`--bs-copper`/`--pf-copper`), как везде в дизайн-системе.
2. **Человеческие подписи** вместо голых тикеров MCFTR/IMOEX — и в легенде графика
   (`BenchmarkChart`, ~строка 1132–1133: `label: "MCFTR (с дивидендами)"` →
   `"Индекс МосБиржи, полная доходность"` с тултипом-пояснением почему именно он
   основной ориентир; `"IMOEX (ценовой)"` → `"Индекс МосБиржи (без дивидендов, справочно)"`),
   и в тексте под графиком (~строка 2683: `"Базовые кривые — полная доходность..."`)
   — сделай короче и понятнее рядовому инвестору: примерно *«Сравниваем с индексом
   МосБиржи полной доходности — это то же самое, что «вложить в рынок целиком, с
   реинвестированием дивидендов». Второй, тонкий — тот же индекс, но БЕЗ дивидендов,
   для справки: разница между линиями — это и есть вклад дивидендов»*.
3. **Смешанный бенчмарк по весам портфеля** — НОВАЯ третья (или переключаемая)
   линия на графике, из `benchmark.sector_blend`. Если `sector_blend` не `null`:
   покажи линию (третий цвет из `--cat-*` палитры, НЕ конфликтующий с существующими
   `--cat-1`/`--cat-8`) с меткой типа «Ваши секторы (если бы просто держали рынок в
   ваших пропорциях)». Обязательно раскрой `sector_blend_coverage_pct` — если <100%,
   явно напиши какие секторы не вошли (`sector_blend_excluded_sectors`) и почему
   (нет отраслевого индекса на эти секторы). **Ключевой инсайт для копирайта** (это
   самая ценная идея этой задачи, донеси её): разница между «Портфель vs MCFTR» и
   «Портфель vs смешанный бенчмарк» разделяет ДВА разных источника результата —
   (а) удачный/неудачный выбор САМИХ СЕКТОРОВ (видно из «смешанный vs MCFTR») и
   (б) удачный/неудачный выбор БУМАГ внутри секторов (видно из «портфель vs
   смешанный»). Например: *«Портфель обогнал MCFTR на X%. Но смешанный бенчмарк (те
   же секторы, что у вас, в рыночных пропорциях) сам по себе обогнал MCFTR на Y% —
   значит часть вашего результата (Y%) объясняется удачным выбором секторов, а
   остальное (X−Y%) — выбором конкретных бумаг внутри них»*. Если пользователю сложно
   визуально сравнить 3 линии — можно вывести и текстовое разложение отдельным
   блоком под графиком.
4. **Отраслевые TR-индексы как пресеты** в «+Добавить сравнение» (~строка 2688–2811,
   там уже есть добавление произвольного тикера/корзины/отношения) — добавь секцию
   быстрого выбора: чипы с названиями секторов (те, что есть в
   `SECTOR_TR_TICKERS` бэка — Нефть и газ/Металлургия/Финансы/Потребительский
   сектор/Транспорт и логистика/Электроэнергетика/Химия/Телеком/IT-сектор/Девелопмент),
   клик — добавляет соответствующий TR-тикер (MEOGTR/MEMMTR/MEFNTR/MECNTR/METNTR/
   MEEUTR/MECHTR/METLTR/MEITTR/MERETR) в `extraSeries` тем же путём, что и ручной ввод
   тикера сейчас (найди, как это делает `TickerInput`/добавление тикера в текущем коде,
   переиспользуй тот же путь данных, не дублируй логику).

## Мелкое: дата цены облигаций/фондов в «Состав»

Для non-equity позиций (`instrument_type: "bond"|"fund"`) в таблице «Состав» покажи
`price_as_of` рядом с ценой — например мелким серым текстом под ценой «на 20.07» —
это честно раскрывает, что цена облигаций/фондов обновляется реже акций (T+1, а не
каждые 5 минут), а не выглядит как «зависшая»/неправильная цифра. Если `price_as_of`
старше ~5 дней — можно чуть выделить (не алармистски, просто заметнее), т.к. это
сигнал реальной проблемы с обновлением, а не обычный лаг.

## Дизайн-конституция (обязательно)

Канон — `styles/basis-design-system.css` (`--bs-*`/`.bs-*` либо локальные проекции
`--pf-*` уже используемые в этом файле). Цвет — только в данных (зелёный/красный для
дельт, медь для акцента интервала/переключателей), хром монохромный. Эпистемические
теги (факт/оценка/суждение) — на КАЖДОМ новом аналитическом утверждении (апсайд к
цели, valuation_flag, CVaR — везде уже используемый паттерн `pf-tag-fact`/
`pf-tag-estimate`/`pf-tag-judgment`, см. `MetricExplainers`). Не создавай новый визуальный
язык для переключателей/чипов — ищи и переиспользуй существующие классы в этом же
файле/`styles/observer-v2.css`/`docs/portfolio-analytics.html` (эталон вида для
Портфеля).
