# B03. Мониторинг: источники, триггеры, алерты, свежесть

**Слой:** методология. Агент без контура обновления протухает (Принцип 7). Этот файл отвечает на три вопроса: ОТКУДА агент узнаёт об изменениях (§2), ЧТО именно и НАСКОЛЬКО меняет каждый тип события (§3), КАК оформляется и журналируется реакция (§4–6).

---

## 1. Ритм контура

- **Ежедневный скан (20–40 мин):** санкционные ленты (OFAC/EU/OFSI), фронтовая сводка, курс и спреды, заявления переговорных треков, лента A/B-источников по watchlist из последнего handoff.
- **Еженедельный дайджест:** дисконты и объёмы экспорта, платёжные прецеденты, аукционы ОФЗ, недельная инфляция, движение по пакетам ЕС/законопроектам США.
- **Ежемесячный пересчёт барометра:** полный проход G1–G13 с публикацией handoff (A02). Привязка к календарю: после данных Минфина по Urals (~5-е число) и отчёта по исполнению бюджета.
- **Внеочередной пересчёт:** любой триггер §3 с пометкой «срочно» ИЛИ совокупный сдвиг оценки ≥0,5 по любому G. Правило: внеочередной барометр может быть частичным (пересчитаны затронутые G + шапка), но обязан обновить stale_after.
- **Квартально:** сверка с бэктест-логом (V01 §7), ревизия списка источников (мёртвые ссылки, деградация качества).

**Правило свежести (синхронизировано с A02):** handoff несёт поле `stale_after` = min(дата+45 дней; ближайшее календарное событие из §2, способное сдвинуть G на ≥0,5 — заседание ЦБ, саммит, дедлайн GL, голосование по пакету). Агент КОМПАНИЯ при протухшем барометре останавливается и запрашивает пересчёт — молча работать на старом запрещено.

## 4. Формат алерта (шаблон)

```
ALERT <дата, время MSK>
Событие: <одна строка>
Источник: <ссылка/издание, класс A/B/C> | Статус: [ФАКТ]/[СЛУХ — требует подтверждения]
Затронуто: G<n> (владелец), смежные: <...>
Предлагаемый шаг: <было → станет>, основание: <строка триггера §3 или «вне таблицы, обоснование…»>
Секторные/компанейские флаги: <тикеры из зоны поражения>
Дедлайн реакции: <внеочередной barometer / 72 ч / плановый>
```

Правило слухов: рынок бежит раньше фактов (уроки K20: ралли на звонке до любых документов) — слух класса C фиксируется алертом со статусом [СЛУХ] и попадает в watchlist, но БАЛЛ не меняет; меняет — подтверждение A/B. Имплайд-реакция рынка на слух фиксируется отдельно (сырьё для C06 §5), это не основание для G-шага.

## 5. Watchlist_30d (обязательный блок handoff)

Формируется на каждом барометре: 5–10 конкретных наблюдаемых событий ближайших ~30 дней с датами (заседание ЦБ, дедлайн GL, голосование по пакету, объявленный раунд переговоров, публикация бюджета) + по каждому: «если исход X → двигаю G так-то». Это превращает мониторинг из реактивного в проверяемый прогноз (T14: каждая строка watchlist — микропрогноз, попадающий в лог V01).

## 6. Журналирование (сырьё для проспективного бэктеста)

Каждый алерт и каждый пересчёт пишутся в лог: дата; событие; было→стало по каждому изменённому G; основание; версия базы (kb_version/commit). Ложные тревоги НЕ удаляются — это данные о калибровке. Раз в квартал лог сверяется по V01 §7: точность триггеров (сколько алертов подтвердилось), скорость (лаг «событие→пересчёт»), пропуски (события, менявшие рынок без нашего алерта, — разбираются поимённо).

**Анти-паттерны контура:** двигать балл по телеграм-каналу; двойной триггер одного события в два G вне парных исключений (§3 №14, №24); «молчаливое подтверждение» старых баллов после stale_after; алерт без секторных флагов (МАКРО обязан думать о потребителе-КОМПАНИИ); удаление ошибочных алертов из лога.
