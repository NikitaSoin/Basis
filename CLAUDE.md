## Продукт
Basis — независимый аналитический слой и «второе мнение» для частного инвестора на
российском рынке (НЕ брокер, НЕ сделки, НЕ «купить/продать»; снижает неопределённость,
не хайп). Ценности: доверие, логика, полнота рассуждений.

Основные разделы:
- **Рынок** — котировки и классы активов (акции, облигации, фьючерсы, фонды,
  валюта/металлы, опционы*) + карточки компаний (глубокий разбор по тикеру).
- **Скринер** — фильтрация и сортировка по готовым метрикам Basis (акции и облигации):
  справедливая цена, P/E, дивдоходность, доходность/риск и т.д. Инструмент поиска —
  выводы за пользователем.
- **Обозреватель** — рыночный фон (лента новостей, макрообзор, отчётности, календари,
  карты рынка, геополитика, ИИ-обзор).
- **Портфель** — состав, агрегирующая таблица, матрица корреляций, ИИ-диагноз,
  стресс-тест.

Планируется / в проработке (часть — заглушки в сайдбаре): стресс-тест портфеля,
ИИ-помощник, сравнение бумаг-аналогов, портфельные стратегии; новые форматы
Обозревателя (утренний/вечерний обзор, «куда инвестировать сейчас», IPO-разбор,
перевод заявлений менеджмента) — кандидаты в ИИ-обзор. Детали и приоритеты — `docs/status.md`.

(*Опционы — НЕ трогать без отдельного ОК владельца.)

Стек: React (CRA, чистый JS; фронт в `/frontend/Basis`, основной UI — `src/App.js`,
**большой файл ~8k строк** + вынесенные компоненты в `src/company/` (FinanceTab.jsx и др.)
и `src/*` (Obs*-разделы Обозревателя)), FastAPI + PostgreSQL + SQLAlchemy + Alembic
(`/backend`), деплой git push → Timeweb (фронт — из закоммиченного `build/`, см. раздел «Готово»).
LLM-прослойка `app/services/llm.py` (провайдер-агностик: DeepSeek/Claude/GPT
переключаются настройкой). Рабочая LLM — DeepSeek для механики; субагенты-аналитики
с суждением — на Opus.

## 🔴 ГОТОВО = РАБОТАЕТ НА ЖИВОМ САЙТЕ
«Код запушен» ≠ готово. Готово = открыл inbasis.ru и УВИДЕЛ результат: данные в БД →
бэк отдаёт → фронт рендерит → задеплоено. Боевые адреса: **фронт `inbasis.ru`**, **API
`nikitasoin-basis-a772.twc1.net`** (адрес API периодически меняется при пересоздании
приложения — если 404/недоступен, сверься с реальным в браузере DevTools→Network).
Деплой не мгновенный: жди смену бандла `main.<hash>.js` / опрашивай боевой эндпоинт (до
10 мин), не суди «сломано» сразу. НИКОГДА не отчитывайся «сделано», если на живом сайте этого нет.
🔴 **ДЕПЛОЙ ФРОНТА = ЗАКОММИЧЕННЫЙ `frontend/Basis/build/`** (pre-built, НЕ пересборка на
сервере). После ЛЮБОЙ правки `src/` ОБЯЗАТЕЛЬНО: `cd frontend/Basis && CI=false npx
--no-install craco build`, затем `git add src build && commit && push`. Иначе изменения НЕ
доедут («ноль изменений на бою» при том, что коммиты идут). Проверка: hash `main.<hash>.js`
в build/asset-manifest.json должен смениться. Из sandbox бой проверяй `curl` с dangerouslyDisableSandbox.
«Готово» по аналитической задаче = сделано ПО МЕТОДИКЕ (включая качественные блоки), а
не «поля заполнены числами». Частично — честно скажи, что да, что нет.

## 🔴 ОТК — СМОТРИ НА ЭКРАН ГЛАЗАМИ, А НЕ НА ЗАМЫСЕЛ
Перед готовностью прогони ОТК по тому, КАК ВЫГЛЯДИТ НА ЖИВОМ САЙТЕ:
- `user-persona-reviewer` — понятно ли, ЧИТАЕТСЯ ли (шрифты, каша, «простыня»),
  отвечает ли «что мне с этим делать». ВЕСЬ экран целиком + сверка с соседними
  вкладками (иконки, переходы, заполненность, единые термины).
- `basis-design-reviewer` — дизайн-конституция (для UI).
Мнение персоны оценивай критично (неполный обзор — не ломай правильное), но РЕАЛЬНЫЕ
дефекты (нечитаемо, простыня, прочерки) чини САМ до возврата.

## 🔴 НЕ ОСТАНАВЛИВАЙСЯ — СООБЩЕНИЕ ВЛАДЕЛЬЦУ = КОНЕЦ РАБОТЫ
Написать владельцу отчёт = завершить ход и встать. Поэтому:
- **Промежуточные чекпойнты — ТОЛЬКО в `docs/work-journal.md`** (что сделано, что ждёт
  ОК, что осталось) — и РАБОТАЙ ДАЛЬШЕ. НЕ пиши промежуточных отчётов в чат: захотел
  отчитаться — это сигнал записать в журнал и продолжить.
- Сообщение владельцу в чат — ТОЛЬКО когда: (а) очередь исчерпана / всё, что можно без
  него, сделано; (б) блокирующая развилка, без которой дальше никак; (в) технический
  предел сессии (контекст). Тогда — сводный отчёт из журнала.
- Развилки НЕ блокирующего характера — в журнал и продолжай очередь.
- UX-баги всего сайта (потеря состояния списков, несогласованные переходы) —
  приоритетнее нового контента.
- Лучше НОВОЕ вчерне, чем простаивать; «вчерне» ≠ без ОТК и ≠ нечитаемо.
- Большие раскатки (сотни единиц) — батчами, коммит после каждого батча, статус в
  журнал, СЛЕДУЮЩИЙ БАТЧ СРАЗУ. Несколько сессий — норма: новая сессия читает журнал и
  продолжает с места обрыва без вопросов.

Очередь и текущий фокус бери из `docs/status.md` (живой), сверху вниз.

## 🟡 ОРГМОДЕЛЬ (сжато)
Ты — гендиректор: понимаешь суть, собираешь команду, отвечаешь за результат на сайте.
Нанимай субагентов адаптивно (`.claude/agents/` по образцу). Масштаб по задаче: мелкая —
сам; средняя — 1 аналитик + ОТК; крупная/незнакомая — продуктовая постановка → данные →
доменный аналитик → реализация → ОТК. Пайплайны — `docs/playbook.md`. Модели:
добытчики/report-fetcher → **Sonnet** (Haiku слабее: хуже находит прямые PDF, сдаётся на
блокировках; на доборе ОДДС Sonnet дал 8/8 vs Haiku 3/4); аналитики с суждением → Opus.
Субагенты НЕ делают git push. 🔴 report-fetcher часто пишет в `sources/extracted_financials.json`
(контракт роли) — ты переносишь в основной `financials.json` (по годам, с выравниванием fiscal_years).

🔴 **КОНТРОЛЬ СУБАГЕНТОВ (беречь лимиты, не зацикливаться):** субагент делает ОДИН проход и
завершается — НЕ крутится бесконечно. В промпте ставь границы: веб-поиск экономно (ориентир
5–8 запросов, не долбить один источник), при нехватке данных ЧЕСТНО ДЕГРАДИРОВАТЬ (null +
data_flag), а не искать до посинения; результат писать ОДНОЙ операцией в конце. Диспетчер (ты)
следит за фоновыми агентами: нормальный прогон — до ~10–15 инструментов / ~10 мин; если агент
идёт заметно дольше или зациклился (повторяет одно и то же) — останови (TaskStop) и перезапусти
с более узкой задачей. НЕ плоди агентов пачками без счёта — батч 3–4 параллельно, дождись, потом
следующий. Раскатка сотнями — по одному батчу за раз, с чекпойнтом в журнал.


## 🎨 ДИЗАЙН (коротко; единый источник — `docs/design-system.md`)
**NEO-система — канон.** Serif заголовки/проза (**Fraunces**) / mono числа (**IBM Plex Mono**,
tabular-nums) / sans UI (Inter). Акцент — **медь** `#C97A4A` (обе темы; сменён с кобальта
по Заданию А владельца, на бою через подмену токенов). Две темы одной логикой: тёплая бумага
«Research Desk» (light) / графит «Command Center» (dark); накатана **алиасингом токенов**
в `.cc-root` (классические имена токенов переопределены на `--cc-*`). Доп. слой — переносимая
ДС `styles/basis-design-system.css` под префиксом `--bs-*`/`.bs-*` (шрифты в public/index.html).
Обязательно ВЕЗДЕ:
- **эпистемические теги** (факт / оценка / модель / суждение) на каждом аналитическом
  утверждении;
- **«4 слоя чтения»**: идентичность → сигнал (одно главное число/вердикт) →
  доказательство + надёжность → действие. **Вердикт поверх данных** — голая таблица без
  интерпретации не считается готовым экраном;
- цвет — ТОЛЬКО в данных; хром монохромный; глиф ▲/▼ у дельт; focus-кольца;
  `prefers-reduced-motion`; никаких «празднующих» анимаций.
Токены НЕ выдумывать — только из `tokens.css`. Дизайн-задачи — `backend/design-tasks/`
(эталон формата там же). Субагенты: `basis-designer` / `basis-design-reviewer`.
Classic-система — легаси за `?classic=1`, для нового не использовать.

### 🔴 РАСКАТКА HTML-ПРОТОТИПОВ ТОЧНО (Обозреватель/Портфель — `docs/*.html`)
Прототипы (`observer-sidebar-v2.html`, `portfolio-analytics.html`) — ОДИН дизайн-язык: те же классы
(`.pill`, `.seg-opt`, `.chip`, `.card`, `.chip-stat`, `.deep-card`), значения и токены (`--copper`,
`--surface-3`, `--ink`). Обозреватель УЖЕ реализован точно в `styles/observer-v2.css` (классы `obs-*`)
+ компоненты `Obs*` в App.js через паттерн `ObserverV2` (тёмный сайдбар, роутинг разделов React-state,
keep-alive). **Портфель делать ТОЧНО ТАК ЖЕ — переиспользовать/копировать `obs-*` классы, не изобретать
через tailwind.** Типовые провалы (были на Обозревателе, чинились через втыки владельца):
0. 🔴🔴 **ГЛАВНОЕ — РАЗМЕТКА ДОСЛОВНО, НЕ «ПО МОТИВАМ».** Причина «код вроде тот же, а на сайте иначе»:
   разметку прототипа АБСТРАГИРУЮТ в свои React-компоненты (`<KpiTile.../>`) и меняют inline-стили на
   tailwind (`tw-text-accent`, `tw-text-[14px]`) → теряется точная сетка (`grid-template-columns:1.15fr 1fr`),
   декор (`border-left:2px solid var(--copper)`), размеры (`clamp(38px,5.2vw,62px)`, `font-size:24px`).
   ПРАВИЛО: КАЖДЫЙ `<div style="...">` прототипа → `<div style={{...}}>` с ТЕМИ ЖЕ значениями; `class=`→
   `className=`; хардкод-число → переменная данных — и ВСЁ. НЕ заворачивать плитки в свои компоненты, НЕ
   подменять inline на `tw-*`. DOM + inline = 1:1 с html → выглядит идентично само. Красный флаг: мало
   `style={{` в компоненте при насыщенном inline-стилями html (у агента было 48 на весь PortfolioV2).
1. **Кнопки/чипы «по мотивам» через tailwind-утилиты** вместо ТОЧНЫХ значений прототипа. Бери дословно:
   `.seg-opt` = `padding:9px 20px; font-weight:700; border-radius:99px`; `.chip` = `padding:8px 16px;
   border-radius:999px`; `.pill` = `border-radius:999px`. Копируй в свой CSS-файл (portfolio-v2.css),
   НЕ пересобирай tailwind-классами.
2. **«Плитки/карта разные», хотя формула та же** — прототип на ИЛЛЮСТРАТИВНЫХ данных (ровные веса), на
   РЕАЛЬНЫХ разброс другой (карта: sqrt(капы) раздувал гигантов → нужен pow 0.15). СВЕРЯЙ ВИЗУАЛЬНО
   результат на реальных данных, а не только код против html.
3. **Правишь не тот компонент** — проверь, что рендерится реально (в Обозревателе `renderFinancials`
   был мёртвым, реальный — FinanceTab.jsx). `grep "return <XTab"` / где подключён в App.js.
4. **CSS в отдельный файл с классами прототипа** (как observer-v2.css), стили через `var(--*)` токены,
   уважать light/dark. По одному разделу за раз, СВЕРКА НА ЖИВОМ САЙТЕ после каждого (не «по коду»).

## ⚙️ Технические правила (кратко; подробности в playbook)
- Backend-фича: модель → схема → сервис → эндпоинт → тест.
- Схема БД — только Alembic-миграции. Массовая закачка — скриптом.
- Секреты — только `.env`, НИКОГДА не хардкодить.
- Простые решения. Три уровня достоверности (факт/оценка/суждение) во всей аналитике.
  Без «купить/продать».
- **Данные и цены:** цена ТОЛЬКО из таблицы `quotes` (живой Тинькофф→БД); `rates.csv` —
  только справочник (тикеры / число акций / дивотсечки), НЕ для цены. Апсайд и
  мультипликаторы считать живьём от цены, не запекать. Числа карточки — из
  `financials.json` (единый источник).
- **Сквозная боль:** устаревшие входные данные (Росстат недоступен машинно, часть фидов
  отстаёт). Помни про свежесть, помечай уровень достоверности, не выдавай старое за факт.
- MOEX ISS и веб — последовательно с паузами, батчами (иначе rate limit).
- «rate limit (not your usage limit)» = повисшая сессия/плотный поток: новая сессия,
  паузы. Новая сессия начинает с чтения `docs/work-journal.md`.
- Объясняй владельцу как НЕ-технарю: видимый эффект → причина → деталь.

## LLM: dev-time vs production
Разовые внутренние задачи на этапе разработки (извлечение/структурирование данных,
оценочные прогоны, разметка контента) — делать ЧЕРЕЗ CLAUDE: сам агент или внутренние
субагенты. НЕ подключать платные сторонние LLM-API (DeepSeek, OpenAI и пр.) для таких
разовых задач — Claude доступен без лимитов по подписке и качественнее; медленнее —
приемлемо, это разовый прогон.

Production-фичи платформы, которым LLM нужен в рантайме под запросы пользователей
(ИИ-обзор, разбор отчёта, интерпретатор, ИИ-диагноз), — отдельный случай: работают через
бэкенд-сервис llm.complete() с провайдером по конфигу. Подписка Claude Code тут неприменима.
Выбор прод-провайдера — продуктовое решение по цене/качеству.
