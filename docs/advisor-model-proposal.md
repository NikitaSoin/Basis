# ТЗ: применение паттерна Advisor к субагентной архитектуре Basis

> Исследовательский документ-рекомендация. НЕ код, НЕ применено. Изменения в
> `.claude/agents/*.md` — только по отдельному ОК владельца.

## 1. Что владелец имел в виду (по описанию + бенчмаркам)

Anthropic раскрыл использование модели "Advisor": быстрый/дешёвый **executor**
(в примере — Sonnet 5) во время генерации может обратиться к более сильному
**advisor** (Fable 5 / Opus) за стратегическим советом — тот читает весь
диалог, даёт план или коррекцию курса, а executor продолжает выполнение сам.
На SWE-bench Pro: Sonnet 5 + Fable-5-Advisor ≈ 92% качества чистого Fable 5
при ≈63% его стоимости; при РЕДКОМ вызове советника (~1 раз на задачу, только
для стратегии) — 96% качества при 46% стоимости.

## 2. Проверено по документации (важная поправка к постановке)

Прочитал первоисточники по обеим ссылкам:

- **Advisor tool** (`platform.claude.com/docs/en/agents-and-tools/tool-use/advisor-tool`) —
  это **бета-инструмент прямого Anthropic Messages API** (заголовок
  `advisor-tool-2026-03-01`). Executor получает `advisor`-tool как обычный
  tool-call: во время генерации может "спросить" advisor-модель, та отвечает
  планом/коррекцией, executor продолжает **в той же сессии**. Годится для
  длинных агентных задач (кодинг-агенты, computer use, multi-step research),
  где большинство шагов механические, а качество плана критично.
- **Managed Agents / multi-agent** (`.../managed-agents/multi-agent`) —
  ОТДЕЛЬНАЯ бета (`managed-agents-2026-04-01`): coordinator-агент делегирует
  work в изолированные **session threads**, каждый субагент — своя модель/
  system prompt/tools/MCP, потоки персистентны (можно дослать follow-up).
  Это инфраструктура "оркестратор + именованные субагенты", не про advisor
  как таковой.

**Ключевой вывод: обе фичи — бета прямого Anthropic API/SDK (Python/TS/Go/…
клиент, `client.beta.agents.create(...)`), НЕ настройка в `.claude/agents/*.md`
Claude Code CLI.** Сегодняшние субагенты Basis (report-fetcher,
financial-analyst и др.) — это markdown-роли, вызываемые через Agent tool
внутри Claude Code сессии; они не используют Anthropic API напрямую и не
имеют доступа к `advisor`-tool или `multiagent`-конфигу как таковым. Прямого
"впилить одну настройку и заработает" пути сегодня нет.

## 3. Что у Basis УЖЕ есть в духе Advisor (без новой инфраструктуры)

Оргмодель CLAUDE.md уже двухуровневая и **функционально** близка к идее
"дешёвый исполнитель + редкий дорогой советник", просто реализована грубее —
через ДВА ОТДЕЛЬНЫХ вызова субагента, а не через tool-call внутри одной
генерации:

- **report-fetcher → Sonnet** (не Haiku: на доборе ОДДС Sonnet дал 8/8 против
  Haiku 3/4 — механическая, но не тривиальная работа, нужен исполнитель
  умнее самого дешёвого уровня).
- **Аналитики с суждением (financial-analyst, governance-analyst,
  macro-analyst, card-critic и др.) → Opus** — весь агент целиком на дорогой
  модели, включая механические шаги (чтение файлов, арифметика, форматирование),
  которые не требуют суждения уровня Opus.

Это уже "экономия через маршрутизацию по роли", но не "экономия через
маршрутизацию ВНУТРИ роли" — там, где аналитик делает 80% механической
сборки данных и 20% реального суждения, всё 100% всё равно идёт по цене Opus.

## 4. Где ПАТТЕРН (не конкретный API) дал бы реальную экономию

Кандидаты — роли с длинным механическим шлейфом и редкими точками
настоящего суждения:

1. **financial-analyst** — самый тяжёлый агент: 5-7 лет отчётности, P/E и
   мультипликаторы, DCF/NAV/SOTP по секторной маршрутизации, сравнение с
   конкурентами. Большая часть — вычисления и сборка по чёткой методичке
   (механика). Настоящее суждение — 2-3 точки: КАКОЙ метод оценки применить
   к сектору (девелопер → NAV, банк → P/BV×ROE и т.п.), и финальная сверка
   "разумна ли получившаяся справедливая цена".
2. **report-fetcher** — навигация e-disclosure→сайт компании→smart-lab,
   парсинг таблиц, проверка арифметики — механика. Редкое суждение: "какой
   из противоречивых источников первичнее" при расхождениях.
3. **governance-analyst / geo-analyst** — сбор фактов (веб-поиск, структура
   владения, санкционные списки) — механика; расстановка баллов по 8
   факторам/сценариев — суждение.

Card-critic, macro-critic, user-persona-reviewer — НЕ кандидаты: это уже
"один вызов = одно целостное суждение" по своей природе, разбивать нечего.

## 5. Рекомендация (без новой инфраструктуры, реализуемо сегодня)

Настоящий `advisor`-tool (mid-generation, общий контекст/кэш одной сессии)
недоступен из Claude Code CLI сегодня. Но его ЭФФЕКТ (дешёвая механика +
редкая дорогая проверка) можно приблизить уже сейчас **ручным двухшаговым
чейнингом**, который я как оркестратор уже умею делать через Agent tool:

- **Шаг 1 (Sonnet, механика):** новая роль-вариант "сборщик" делает то же,
  что сейчас делает Opus-версия агента, до точки суждения — считает
  мультипликаторы/DCF по формуле, собирает факты, НЕ выбирает метод и НЕ
  формулирует финальный вывод.
- **Шаг 2 (Opus/Fable, редкий советник):** отдельный короткий вызов получает
  СЖАТЫЙ результат шага 1 (не сырые данные, а уже посчитанное) и делает
  ровно то, что требует суждения: выбор метода/маршрутизация, сверка на
  разумность, финальная формулировка.

Это не идентично API-паттерну (нет общего кэша контекста, нет "advisor
видит весь диалог executor'а мид-дженерейшн" — видит только сжатую передачу
между двумя отдельными вызовами), но даёт похожий экономический эффект:
большая часть токенов генерируется дешёвой моделью, дорогая тратится только
на точку реального суждения.

**Пилот, ПРЕЖДЕ чем менять `.claude/agents/*.md` для всех ролей:**
прогнать financial-analyst как двухшаговый чейн на 3-5 компаниях (разных
секторов — банк/девелопер/сырьё, чтобы задеть разную маршрутизацию методов),
сравнить с текущим цельным Opus-прогоном по (а) качеству итоговой оценки
(субъективно + фактчек), (б) стоимости/времени. Если лифт экономики
подтвердится без потери качества — расширять на report-fetcher/
governance-analyst/geo-analyst по той же схеме; если качество просядет —
паттерн не переносится на "суждение, растворённое по всей задаче", а не
сосредоточенное в 1-2 точках, и эксперимент честно закрывается.

**Отдельно проверить при следующей возможности:** появится ли `advisor`-tool
или `multiagent`-конфиг в самом Claude Code CLI/agent-конфигурации (не
только в сыром Anthropic API) — тогда пилот выше станет не нужен, платформа
даст это "из коробки" с более сильным эффектом (общий контекст/кэш).

## 6. Что НЕ делать сейчас

- Не строить отдельную интеграцию с сырым Anthropic API (`client.beta.agents.create`)
  в обход Claude Code CLI ради этой задачи — крупная инженерная инвестиция
  при неподтверждённом ROI для Basis (весь текущий пайплайн аналитиков живёт
  внутри Claude Code сессий, не как отдельный сервис).
- Не трогать `.claude/agents/*.md` без отдельного ОК владельца и без
  результатов пилота выше.
