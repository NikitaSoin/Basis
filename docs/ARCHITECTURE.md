# Карта проекта Basis (inbasis.ru)

> Документ для владельца продукта. Простым языком: из чего состоит проект,
> где какие данные лежат и как всё связано. Обновлён: 2026-06-05.

---

## Общая картина в трёх предложениях

**Фронтенд** (то, что видит пользователь) — одно React-приложение, почти весь
интерфейс живёт в одном большом файле `App.js`. **Бэкенд** (сервер) — Python
(FastAPI), он отдаёт фронтенду данные двумя способами: часть из базы данных
PostgreSQL (список компаний, котировки, портфели), а вся аналитика по
компаниям (бизнес-модель, финансы, геополитика и т.д.) — из готовых файлов
в папке `backend/companies/<ТИКЕР>/`. **Деплой** — пуш в GitHub, Timeweb сам
забирает изменения и пересобирает сайт за 3–10 минут.

```
Пользователь → inbasis.ru (React-фронт на Timeweb)
                  │  запросы /api/...
                  ▼
        Бэкенд FastAPI (Timeweb, nikitasoin-basis-e726.twc1.net)
                  │                         │
                  ▼                         ▼
        PostgreSQL (компании,      Файлы backend/companies/<ТИКЕР>/
        котировки, портфели,       (business_model.md, financials.json,
        пользователи)              geo.json и др. — пишут ИИ-агенты)
```

| Таблица | Зачем | Поля |
|---|---|---|

| **users** | Пользователи | id, email, hashed_password (пароль в зашифрованном виде), is_active, subscription_type, subscription_expires_at, created_at |
| **portfolios** | Портфели пользователей | id, user_id, name, description, created_at |

> **Важно про данные двух типов.** В БД — только «живое»: список компаний,
> котировки, пользователи, портфели. Вся глубокая аналитика (7 блоков карточки)
> — это **файлы** в `backend/companies/<ТИКЕР>/`, их создают ИИ-субагенты
> (business-model-analyst, financial-analyst, governance-analyst,
> market-analyst, macro-analyst, geo-analyst), и они попадают на сайт через
> обычный git push — без БД.

## 4. Как устроен деплой

Цепочка: **git push → GitHub → Timeweb видит новый коммит → пересборка →
через 3–10 минут новая версия в бою.** Фронт и бэк — два отдельных
приложения на Timeweb, собираются независимо.

| Файл | Роль |
|---|---|
| `.github/workflows/deploy.yml` | **Пустышка** (заглушка «Deploy step removed»). GitHub Actions ничего не деплоит — деплой делает сам Timeweb, который подписан на репозиторий. |
| `backend/Dockerfile` | Рецепт сборки бэкенда: образ Python 3.11-slim → установить зависимости из `requirements.txt` → запустить сервер. Без лишних системных пакетов (если сборка падает на apt-get/DNS — это сбой платформы Timeweb, не наш код; лечится перезапуском сборки). |
| `backend/start.sh` | Скрипт старта, устойчивый к проблемам с БД: 1) сервер поднимается **сразу** и отвечает на `/api/health` даже без БД; 2) **миграции Alembic применяются в фоне** с 30 повторами — именно так изменения структуры БД и данные (сектора, пары ао/ап) доезжают до боя автоматически; 3) если миграции не прошли — сервер живёт, а не падает в цикл перезапусков. |
| `backend/import_data.sh` | **Ручной** разовый импорт: наполняет таблицы `companies` (из rates.csv) и `company_profiles` (из JSON-файлов). Запускается только когда нужно, не при каждом старте. |

Проверка после пуша (правило из CLAUDE.md): подождать сборку и убедиться, что
бой реально обновился (для бэка — опрашивать эндпоинт до появления новых
данных, для фронта — что сменилось имя бандла `main.<hash>.js`), и только
потом судить о результате.

## 5. Схема связей: что на экране ← какой эндпоинт ← где данные

**Экран «Рынок» (список секторов и компаний):**
```
CompaniesView → GET /api/companies        → таблицы companies + quotes
             → GET /api/quotes/realtime   → MOEX ISS / Tinkoff (живые цены)
```
Сектор компании — поле `companies.sector`; полная капитализация — сумма
ао+ап через `paired_ticker`.

**Карточка компании, по вкладкам:**
```
Обзор            → GET .../profile             → таблица company_profiles
                 → GET /api/companies/{id}/analysis → таблица company_analyses
Бизнес-модель    → GET .../business-model      → файл business_model.md
Финансы          → GET .../financials(-summary)→ файлы financials.json/.md
                 → GET /api/sectors/{s}/peers  → файл sectors/<сектор>/peers.json
Корп. управление → GET .../governance(-summary)→ файлы governance.json/.md
Рынки            → GET .../market(-summary)    → файлы market.json/.md
Макро            → GET .../macro(-summary)     → файлы macro.json/.md
Геополитика      → GET .../geo(-summary)       → файлы geo.json/.md
```

**Портфель:**
```
PortfolioView → GET/POST /api/portfolios(+/positions) → таблицы portfolios,
                portfolio_positions (+ companies и quotes для цен и расчётов)
```

**Обозреватель:**
```
OverviewView → GET /api/market/overviews → таблица market_overviews
             → POST /api/market/overviews/generate → Claude API → в ту же таблицу
```

**Вход/профиль:**
```
AuthModal/ProfileView → POST /api/register|login, GET /api/me → таблица users
```

## 6. Работа с базой данных: чтение, изменение и две БД (локальная ↔ Timeweb)

### Главное, что нужно понимать

Баз данных действительно **две**, и они **НЕ синхронизируются между собой
напрямую**. Никакого «провода» между твоим маком и Timeweb нет:

| | Локальная БД | Боевая БД |
|---|---|---|
| Где живёт | На твоём маке (PostgreSQL через Homebrew) | На серверах Timeweb, рядом с боевым бэкендом |
| Кто к ней подключается | Бэкенд, запущенный локально, и скрипты с мака | Только боевой бэкенд на Timeweb |
| Зачем нужна | Разработка и проверка: «прогнать миграцию, посмотреть, что получится» | Реальные данные сайта: пользователи, портфели, котировки |
| Адрес задаётся | Файлом `backend/.env` (`DATABASE_URL=postgresql://postgres@localhost:5432/invest_db`) | Переменной окружения `DATABASE_URL` в настройках приложения на Timeweb |

Код везде один и тот же: он читает адрес БД из переменной `DATABASE_URL`
(`backend/app/db/session.py`). На маке переменная указывает на локальную базу,
на Timeweb — на боевую. **Один код — две базы, выбор через настройку.**
Для нелокальных адресов код сам добавляет шифрование соединения (`sslmode`).

С мака до боевой БД напрямую не достучаться (и это правильно — безопасность).
Единственный «мост» между базами — **git-репозиторий**: всё, что должно
измениться в боевой БД, едет туда в виде кода (миграций), а не данных.

### Как данные ЧИТАЮТСЯ (каждый запрос пользователя)

```
Пользователь открыл «Рынок»
→ фронт зовёт GET /api/companies
→ эндпоинт берёт сессию БД (get_db в app/db/session.py)
→ сервис company.py делает SELECT из таблиц companies и quotes
→ отдаёт фронту JSON
```
Каждый запрос получает свою короткую сессию подключения: открыл — прочитал —
закрыл. Боевой бэкенд читает боевую базу, локальный — локальную.

### Как данные ПОЯВЛЯЮТСЯ и ОБНОВЛЯЮТСЯ в БД — четыре канала

**Канал 1. Автоматика внутри сервера (котировки).**
Планировщик (`quotes_updater.py`) сам ходит на MOEX и пишет свежие цены
в таблицу `quotes`: в торговые часы — каждые 5 минут, вечером — раз в час,
в выходные — раз в 6 часов. Никто ничего не запускает руками. Работает
в каждой среде со своей базой (на бою — с боевой).

**Канал 2. Действия пользователей на сайте.**
Регистрация, создание портфеля, добавление позиции — это POST-запросы,
эндпоинты пишут в таблицы `users`, `portfolios`, `portfolio_positions`.
Эти данные существуют ТОЛЬКО в боевой базе (локально их нет и не нужно).

**Канал 3. Миграции Alembic — главный способ менять боевую БД.**
Миграция — это файл-инструкция в `backend/migrations/versions/`:
«добавь колонку», «исправь сектора у 15 компаний», «удали дубль RHYD»,
«проставь пары ао/ап». Путь изменения всегда такой:

```
1. Пишем файл миграции на маке
2. Проверяем на ЛОКАЛЬНОЙ базе:  alembic upgrade head
3. git push → GitHub → Timeweb пересобирает бэкенд
4. start.sh при старте сам выполняет alembic upgrade head НА БОЕВОЙ базе
   (в фоне, с 30 повторами — см. раздел «Деплой»)
5. Боевая БД изменена. Руками на сервере делать ничего не нужно.
```

Это и есть ответ на вопрос «как связаны две БД»: **они связаны через
одинаковую историю миграций**. Alembic хранит в каждой базе отметку
«какая миграция применена последней» (служебная таблица `alembic_version`)
и при старте доводит базу до актуальной версии. Поэтому локальная и боевая
базы имеют одинаковую СТРУКТУРУ и одинаковые СПРАВОЧНЫЕ данные (сектора,
пары тикеров), но разное «живое» наполнение (пользователи, свежие котировки).

**Канал 4. Ручной импорт больших данных (`import_data.sh`).**
Разовая операция «наполнить базу с нуля»: загрузить 263 бумаги из
`data/rates.csv` в `companies` и 262 профиля из JSON-файлов в
`company_profiles`. Запускается вручную в консоли того окружения, чью базу
наполняем (для боя — в консоли приложения на Timeweb). Импорт аккуратный:
существующие записи не затирает (дубли пропускает, профили обновляет
по тикеру).

### Почему файлы аналитики НЕ требуют возни с БД

Семь блоков карточки (бизнес-модель, финансы, …, геополитика) — это файлы
в `backend/companies/<ТИКЕР>/`, которые едут на бой обычным `git push`
внутри образа бэкенда. БД не участвует вообще. Поэтому раскатка, например,
262 гео-блоков не требовала ни миграций, ни импорта — только пуш и деплой.

### Шпаргалка: «я хочу изменить X — что делать?»

| Хочу изменить | Канал | БД трогаем? |
|---|---|---|
| Текст/цифры в блоке аналитики компании | Правим файл в `backend/companies/`, git push | Нет |
| Сектор компании, название, пары ао/ап | Файл миграции → проверка локально → git push | Да, через миграцию |
| Структуру таблиц (новая колонка/таблица) | То же: миграция → локально → push | Да, через миграцию |
| Котировки | Ничего — обновляются сами | Сами |
| Список компаний с нуля / профили | `import_data.sh` в консоли нужного окружения | Да, вручную |
| Данные пользователей | Никогда руками — только через сайт/эндпоинты | — |
